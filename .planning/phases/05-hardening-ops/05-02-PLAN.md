<objective>
Refactor the daily and weekly report jobs to query Meta and GA4 data in independent guarded blocks, so a failure in one source's data fetch does not abort the other source's report section. Extend the report builder to accept availability flags and inject human-readable notices when a source is unavailable.

Purpose: Currently a single try/except wraps all data queries — one exception aborts the whole report. This satisfies success criterion SC-2: per-source failures degrade gracefully with explicit "data unavailable" notices rather than silent drops.

Output: 4 modified/created files. No new dependencies.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB query result → report HTML | Query results enter the builder; dynamic content is html.escape()'d in the builder |
| ingestion_log.status → availability flag | Status string from DB read; used only for boolean comparison ("failed") — not rendered |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Information Disclosure | Error message in logger.warning | accept | `error=str(exc)` logged only to structlog — not included in Telegram report HTML |
| T-05-02-02 | Tampering | ingestion_log.source SQL param | mitigate | Named params (`:source`) — hardcoded string literal "meta_ads" / "ga4" passed as param value; no user input reaches this query |
| T-05-02-03 | Denial of Service | Double-query per source (data + ingestion_log) | accept | ingestion_log has at most a few hundred rows; the DESC LIMIT 1 query with the indexed `started_at` column is O(log n); no performance impact |
| T-05-02-04 | Information Disclosure | "data unavailable" notice in Telegram report | accept | Notice contains no exception details or stack traces — only a static operational message visible to allowlisted operators |
</threat_model>
